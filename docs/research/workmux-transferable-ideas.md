# Transferable ideas from `raine/workmux`

Research date: 2026-07-24
Upstream snapshot: [`raine/workmux@fe2a224`](https://github.com/raine/workmux/tree/fe2a2249e6fa12bac7ab495d85c132786b90f2c5)

## Executive view

Workmux is a terminal/worktree multiplexer; local-agent-bridge is a durable, provider-neutral control plane. The bridge should not replace its worker supervisor, collaboration ledger, exact-head review, or merge train with tmux. The useful ideas are the human-control and workspace-lifecycle patterns around that core.

The bridge already exceeds workmux in several areas: durable cross-app collaborations, self-contained writer Git custody, issue claims, role-aware provider concurrency, protected coordinator wakes, exact-head GitHub review, and serialized conflict-aware integration. The gaps are mostly operational ergonomics: seeing useful live evidence, acting on a selected lane, avoiding duplicate starts, preparing workspaces consistently, and retiring workspaces safely.

## Recommended adaptations

| Priority | Idea | What workmux demonstrates | What the bridge has today | Bridge-shaped adaptation |
| --- | --- | --- | --- | --- |
| P0 | Turn-aware native activity | Codex fires hooks for both a parent and its subagents in the same pane. Workmux keys active turns by `session_id + turn_id` and stays `working` until the active set is empty ([implementation](https://github.com/raine/workmux/blob/fe2a2249e6fa12bac7ab495d85c132786b90f2c5/src/state/codex_status.rs#L1-L15), [state transition](https://github.com/raine/workmux/blob/fe2a2249e6fa12bac7ab495d85c132786b90f2c5/src/state/codex_status.rs#L178-L202)). | [`scripts/host-activity-hook.mjs`](../../scripts/host-activity-hook.mjs) reads only a session/thread ID, while [`src/host-activity-store.mjs`](../../src/host-activity-store.mjs) stores one active bit per provider session. A subagent stop can therefore hide a still-running parent turn in Mission Control. | Persist active run IDs per native session. On start/working, add `turn_id` (and retain agent metadata for diagnosis); on subagent stop, remove only that run; on authoritative root stop, clear the set. Render the lane active while any run remains. Keep a compatibility path for providers whose hooks do not expose turn IDs. |
| P0 | Actionable Mission Control | Its dashboard can preview terminal output, send input, jump to an agent, open a PR/checks, kill a process, and sweep worktrees ([dashboard guide](https://github.com/raine/workmux/blob/fe2a2249e6fa12bac7ab495d85c132786b90f2c5/docs/guide/dashboard/index.md)). | [`src/mission-control.mjs`](../../src/mission-control.mjs) is deliberately read-only; control exists in separate broker tools. | Add an action palette backed by broker operations: open PR/checks, copy collaboration/worktree ID, acknowledge a processed wake, answer and continue, retry/recover, or request cancellation. Every action should carry lane ID plus revision and return a durable receipt. Do **not** inject raw keystrokes into provider processes. |
| P0 | Independent activity pulse | Workmux distinguishes hook status from actual pane activity and marks a supposedly working agent interrupted when pane output stops changing; dashboard and sidebar share the result ([status guide](https://github.com/raine/workmux/blob/fe2a2249e6fa12bac7ab495d85c132786b90f2c5/docs/guide/status-tracking.md)). | The bridge distinguishes heartbeat freshness from narrative freshness, but a live heartbeat can still accompany a stale narrative. | Have the supervisor record privacy-preserving progress counters: stdout/stderr byte count, last-output timestamp, tool-event count, and last provider progress event. Show `alive / output quiet / narrative stale` separately. Use this as diagnosis, not automatic failure: long inference can be legitimately quiet. |
| P1 | Idempotent resume-or-start | `add --open-if-exists` attaches to an existing worktree rather than creating a duplicate, and `--dry-run` prints the resolved plan ([add reference](https://github.com/raine/workmux/blob/fe2a2249e6fa12bac7ab495d85c132786b90f2c5/docs/reference/commands/add.md)). | Issue claims prevent some duplicate writers, but a chair can still create overlapping review/publication collaborations and stale duplicate records. | Add `resume_or_start` with a stable key such as repository + issue/PR + exact head + role + workspace. Return the compatible live/recoverable collaboration when one exists; reject incompatible ownership; otherwise create. Provide a dry-run explaining the match decision. |
| P1 | Declarative workspace recipes | Global/project configuration can copy or link selected files and run `post_create`, `pre_merge`, and `pre_remove` hooks with explicit lifecycle environment variables ([configuration guide](https://github.com/raine/workmux/blob/fe2a2249e6fa12bac7ab495d85c132786b90f2c5/docs/guide/configuration.md)). | [`src/writer-checkout.mjs`](../../src/writer-checkout.mjs) safely creates self-contained writer checkouts, but repository bootstrap commands and cache preparation are not one first-class, inspectable recipe. | Add project-owned checkout recipes: `postCreate`, `prePublish`, `preRetire`, and explicit file/cache operations. Resolve them before provider launch, show them in preflight, record outcomes, and support re-sync. Default-deny secret copying; prefer shared package-manager/build caches over symlinked mutable `node_modules`; keep verification policy separate from setup. |
| P1 | Recoverability-proven workspace sweep | Workmux identifies merged/closed PRs, deleted upstreams, and locally merged branches; dirty worktrees are not selectable in its bulk sweep ([dashboard sweep](https://github.com/raine/workmux/blob/fe2a2249e6fa12bac7ab495d85c132786b90f2c5/docs/guide/dashboard/index.md#sweep)). Its cleanup implementation first renames a worktree to a trash path, then prunes metadata and branches, then deletes the trash directory ([cleanup source](https://github.com/raine/workmux/blob/fe2a2249e6fa12bac7ab495d85c132786b90f2c5/src/workflow/cleanup.rs)). | `bridge cleanup` safely archives terminal control-plane state and intentionally never removes worktrees or branches. | Add a separate `bridge workspace sweep` that is dry-run by default. Require no live lease/worker, a clean tree, no unpushed commits, terminal collaboration/portfolio state, and an exact-SHA recovery source on GitHub or another retained ref. Quarantine by atomic rename before final deletion, recheck immediately before each mutation, and keep branches unless their recovery proof is recorded. Never broaden `bridge cleanup` itself. |
| P2 | Fleet recovery plan | `workmux resurrect --dry-run` reconciles persisted agent records with live worktrees, skips already-open/main worktrees, chooses resume behavior per agent, and consumes stale state only after successful restoration ([resurrect reference](https://github.com/raine/workmux/blob/fe2a2249e6fa12bac7ab495d85c132786b90f2c5/docs/reference/commands/resurrect.md), [planner source](https://github.com/raine/workmux/blob/fe2a2249e6fa12bac7ab495d85c132786b90f2c5/src/workflow/resurrect.rs)). | The machine supervisor adopts live workers and `bridge recover` handles one collaboration, but there is no operator-friendly fleet restore plan after a crash. | Add `bridge recover --all --dry-run`: classify each lane as adopt, resume provider session, wait for coordinator, skip already live, protected `needs_user`, or indeterminate. Execution should be opt-in and revision-fenced; it must never restart terminal or ambiguous lanes automatically. |
| P2 | One resolved configuration view | Workmux layers XDG global config with nearest project config, provides explicit list-composition semantics, and exposes `config path`, `reference`, and initialization ([configuration guide](https://github.com/raine/workmux/blob/fe2a2249e6fa12bac7ab495d85c132786b90f2c5/docs/guide/configuration.md)). | The bridge intentionally separates provider, GitHub App, fallback, concurrency, and local-model JSON plus environment overrides; `doctor` diagnoses the result but configuration discovery is scattered. | Keep security domains separate on disk, but add `bridge config effective --workspace … --explain`, `config paths`, and `config reference`. Report every resolved value's source, redacting secrets. If project overrides are added, define replace-vs-append semantics explicitly rather than deep-merging arrays implicitly. |
| P2 | Conflict workspace UX | Workmux records the base branch used at creation, has a dedicated rebase command, and leaves conflicts in place with recovery instructions rather than cleaning up underneath the user ([rebase reference](https://github.com/raine/workmux/blob/fe2a2249e6fa12bac7ab495d85c132786b90f2c5/docs/reference/commands/rebase.md), [merge reference](https://github.com/raine/workmux/blob/fe2a2249e6fa12bac7ab495d85c132786b90f2c5/docs/reference/commands/merge.md)). | [`src/merge-train.mjs`](../../src/merge-train.mjs) already has stronger exact-head validation and arbitration dossiers, but the resolution workspace is mostly a coordinator concern. | Expose `bridge conflict inspect/open/continue/abort` for the one authorized resolution writer. Show target/head/base SHAs, conflicted paths, dossier, and next safe command; preserve the checkout until a replacement exact head is reviewed or the arbitration is explicitly abandoned. |
| P2 | Human handles and event waits | Workmux addresses agents with stable handles, supports `project:handle` disambiguation, and provides `wait --any` with distinct completion, timeout, missing, and crash outcomes ([coordinator skill](https://github.com/raine/workmux/blob/fe2a2249e6fa12bac7ab495d85c132786b90f2c5/skills/coordinator/SKILL.md#L14-L27), [wait contract](https://github.com/raine/workmux/blob/fe2a2249e6fa12bac7ab495d85c132786b90f2c5/skills/coordinator/SKILL.md#L84-L101)). | The bridge uses durable UUIDs and has a portfolio-specific race-safe waiter, but UUID-heavy Mission Control rows and repeated polling are awkward for people and simple scripts. | Keep UUIDs canonical while adding aliases such as `nolvaren-next:#183:writer` and `thriftybite:PR-211:review`. Add a generic event wait over aliases or IDs with `--any`, explicit terminal reasons, and a cursor so callers wake once without replaying old payloads. |

## Implementation order

1. **Turn-aware native activity, Mission Control action palette, and activity pulse.** These directly fix false-idle reporting and reduce the black-hole experience without changing scheduling or merge authority.
2. **`resume_or_start`.** Prevent duplicate lanes before adding more ways to control them.
3. **Workspace recipes.** Reduce cold-start time and repository-specific setup mistakes while retaining self-contained Git custody.
4. **Workspace sweep.** Build it as a separate recoverability-first subsystem after recipes and lane identity are stable.
5. **Fleet recovery, stable handles/event waits, resolved config, and conflict UX.** These compound the earlier primitives but are less urgent.

## Patterns not to copy literally

- **Do not make tmux the source of truth.** Workmux can equate pane/window lifecycle with task lifecycle; the bridge spans desktop apps, CLIs, detached workers, and provider session APIs. Tmux should be an optional presentation adapter at most.
- **Do not send unstructured keystrokes to a delegated provider.** Workmux input forwarding is convenient for a human-owned pane, but broker actions need schemas, authorization checks, revision fencing, and receipts.
- **Do not clear protected attention merely because the user focused a view.** Workmux auto-clears waiting/done icons on focus; a bridge wake should clear only after the coordinator processes and acknowledges it.
- **Do not age-kill workers from status timestamps alone.** `workmux reap-agents` is dry-run by default and escalates through `Ctrl-C`/`Ctrl-D` ([command reference](https://github.com/raine/workmux/blob/fe2a2249e6fa12bac7ab495d85c132786b90f2c5/docs/reference/commands/reap-agents.md), [implementation](https://github.com/raine/workmux/blob/fe2a2249e6fa12bac7ab495d85c132786b90f2c5/src/command/reap_agents.rs)). The bridge must additionally prove lease ownership, process identity, and absence of publication/cleanup operations.
- **Do not adopt local merge-and-delete as the normal path.** The bridge's GitHub PR, exact-head review, hosted CI, builder-App merge, and recovery-proof requirements are intentionally stronger than workmux's local `merge` lifecycle.

## Small, low-risk experiments

1. Add a read-only transcript/event preview and `open PR/checks` actions to Mission Control.
2. Add supervisor output counters and show a three-part freshness line: process, output, narrative.
3. Add `bridge start --dry-run --resume-if-compatible` before changing the MCP schema.
4. Prototype `bridge workspace sweep` as a report only; compare its classifications with manual cleanup decisions for several weeks before enabling `--apply`.
